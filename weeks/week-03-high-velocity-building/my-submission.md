# My Submission: Week 03

## Submission summary

This week I did not ship another independent iteration of the Open Source Readiness Field Guide.

Instead, I reached a pivot point.

After learning about [Eclipse OSILK](https://projects.eclipse.org/projects/technology.osilk) and speaking with people connected to that ecosystem, I decided that the **Open Source Readiness Field Guide** should be shelved as an independent project. The problem space still matters, but creating a competing resource would not be the best use of the remaining DevRel University project time.

The better path is:

1. Stop building the **Open Source Readiness Field Guide** as a standalone artifact.
2. Explore **OSILK** as the more mature existing project in the same space.
3. Look for ways to contribute, give feedback, or help strengthen the ecosystem around that project.
4. Pivot my DevRel University build focus toward a smaller product-shaped idea: **PlainTrack**.

## Part 1: Thinking beyond the thing itself

### Original project: Open Source Readiness Field Guide

For the original project, several homework questions are now partially obsolete because the project is no longer moving forward as a standalone build.

| Homework question | Answer after pivot |
| --- | --- |
| How will people learn about the project? | I should not try to make people learn about a competing standalone field guide. A better path is to point people toward **OSILK** and learn where additional explanation, contribution, or DevRel support may help. |
| How will people learn how to use it? | For the shelved project, this becomes obsolete. If I contribute to **OSILK**, usage education should align with OSILK's existing training-as-code and module structure. |
| How will I attract the right people? | I should not attract people to an unnecessary parallel project. I can instead help create gravity around existing open source readiness learning resources. |
| How could I gather feedback? | Feedback should come from **OSILK** maintainers, open source office practitioners, SMEs, and people who would actually use open source readiness learning material. |
| What would make someone continue using or following it? | Trust, currency, modularity, practical examples, and confidence that the material is connected to real open source practice. Those are better pursued through an existing ecosystem than a new solo artifact. |
| What other parts are still needed? | Before creating anything new, I need a gap map: what already exists, what is missing, and where I can contribute without duplicating work. |

### Emerging project: PlainTrack

**PlainTrack** is the product-shaped idea I am considering for the remaining program time.

Working problem statement:

> Work-time records are organizationally important, but most tools make the source of truth opaque, platform-dependent, and disconnected from developer workflows. PlainTrack solves the gap between relevant work-time bookkeeping and developer-native, file-based workflows.

Early answers to the homework questions:

| Homework question | PlainTrack working answer |
| --- | --- |
| How will people learn about the project? | Through a concise GitHub README, a short public explanation, a practical example workflow, and direct sharing with a known group of potential users. |
| How will people learn how to use it? | Through a minimal tutorial that starts with a plain-text or file-based example, then shows how the record can be reviewed, versioned, and exported. |
| How will I attract the right people? | By speaking directly to developers or technical teams who dislike opaque platform-centered time tracking and prefer workflows close to files, Git, or command-line habits. |
| How could I gather feedback? | Through direct conversations with possible users, compliance stakeholders, and a small prototype walkthrough. |
| What would make someone continue using it? | Trust in the source of truth, low friction, transparent records, easy review, and confidence that the workflow does not create compliance problems. |
| What other parts are still needed? | MVP scope, example format, compliance disclaimer, README, installation or usage path, demo, and a simple feedback channel. |

## Part 2: Ship another iteration

### What did I build or improve this week?

I shipped a strategic project decision rather than a code feature.

The meaningful iteration this week was:

* recognizing that the original project overlaps with a stronger existing project
* deciding not to build a competing resource
* reframing my contribution path toward the existing **OSILK** ecosystem
* reopening a smaller product idea that can move into actual building mode
* starting to frame **PlainTrack** around audience, problem, and first validation needs

### What feedback influenced the decision?

The decision was influenced by human network feedback, not only AI-assisted research.

A conversation with a friend from the Eclipse Foundation led me to **OSILK**. A deep dive with the project lead showed that the overlap was real enough to treat **OSILK** as the better existing gravitational center.

This connects directly to Week 02's lesson about growing your network before you need it. I did not enter Week 03 looking for this project decision, but the network surfaced information I would not have found as quickly on my own.

### What part took the longest?

The longest part was accepting the decision.

The project still felt meaningful, and shelving it created the uncomfortable question of whether I was losing momentum halfway through the program. But continuing simply because I had already invested time would have been a sunk-cost decision.

### Where did AI help accelerate?

AI helped with:

* structuring the reflection
* comparing possible project directions
* turning rough notes into documentation
* thinking through audience, onboarding, feedback, and distribution questions
* testing Week 03 concepts such as METT-TC, gravity, and knowledge saturation to the pivot

### Where did I still rely on human judgment?

Human judgment was required for the actual decision.

AI could help organize the evidence, but it could not decide whether creating a competing project was appropriate. That required context, values, trust, and a sense of what kind of contributor I want to be.

This is the central Week 03 lesson in practice: I can outsource parts of the thinking process, but I cannot outsource my understanding.

## Connection to Week 02

Week 02 introduced several ideas that became practical in Week 03:

* Pivots are normal.
* Time spent on a path is not proof the path is still right.
* Your network matters before you urgently need it.
* Build and learn in public.
* The goal is not to predict the future, but to create optionality.

The shelving decision is a direct application of those ideas.

## Connection to Week 03

Week 03 added the systems lens.

The pivot can be understood through several concepts from the session:

* **Snipers both eyes open:** I needed to see the ecosystem, not just the artifact I wanted to create.
* **The system imperative:** Regular documentation and reflection helped make the decision visible.
* **Optimal knowledge saturation:** More independent research was less valuable than deciding whether to contribute or pivot.
* **Gravity:** OSILK already has ecosystem gravity; I should explore how to support it rather than fragment it.
* **METT-TC:** The next project needs a clearer mission, target market, support system, timeline, and cost boundary from the beginning.
