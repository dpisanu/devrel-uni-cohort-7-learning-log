Week 03 of the #DevRelUni journey came with an 𝚞̲𝚗̲𝚎̲𝚡̲𝚙̲𝚎̲𝚌̲𝚝̲𝚎̲𝚍̲ project turn.

My project so far had been the 𝗢𝗽𝗲𝗻 𝗦𝗼𝘂𝗿𝗰𝗲 𝗥𝗲𝗮𝗱𝗶𝗻𝗲𝘀𝘀 𝗙𝗶𝗲𝗹𝗱 𝗚𝘂𝗶𝗱𝗲:
> A practical orientation resource for small and midsize companies trying to better understand open source usage, responsibility, compliance, sustainability and adjacent software supply chain topics.

Then my network did what networks sometimes do: It opened a door I did not know I needed. Thanks, @𝗖𝗵𝗿𝗶𝘀𝘁𝗶𝗮𝗻.

Through a conversation with Christian, I learned about an existing Eclipse project called 𝗢𝗦𝗜𝗟𝗞.

After a follow-up project deep dive with the project lead, @Florent Zara, it became clear that 𝗢𝗦𝗜𝗟𝗞 already addresses much of the same space with more maturity than I could realistically build during this short program.

So I had to take a hard decision for my project as a standalone effort.
Not delete it. Not call it a failure. Shelve it.

And this is where Week 02 came back unexpectedly.

In 𝗡𝗮𝗱𝗲𝗿 𝗗𝗮𝗯𝗶𝘁’s session, we talked about sunk cost fallacy as a career and project risk.

I did not expect that lesson to show up this quickly, but it did.

The question became: 𝘋𝘰 𝘐 𝘸𝘢𝘯𝘵 𝘵𝘰 𝘣𝘶𝘪𝘭𝘥 𝘢 𝘤𝘰𝘮𝘱𝘦𝘵𝘪𝘯𝘨 𝘳𝘦𝘴𝘰𝘶𝘳𝘤𝘦 𝘧𝘳𝘰𝘮 𝘴𝘤𝘳𝘢𝘵𝘤𝘩?

The honest answer: 𝙽̲𝚘̲.

The better path is to explore how I can support, learn from, or contribute to the existing project, and shift my DevRel University build energy toward a smaller product-shaped idea that I can validate faster.

That part hurt a little. But it also felt like the right decision.

The takeaway for me: Sunk cost fallacy is real. It can hurt. But earlier is better than later.

I’ll keep documenting the pivot on GitHub as part of my learning log.
