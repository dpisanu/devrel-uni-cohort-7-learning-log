# Activities: Week 03

## Activity context

The Week 03 activity focused on thinking beyond the core product.

Instead of only asking "What are you building?", the activity asked how a project creates the surrounding experience that helps people discover it, understand it, try it, give feedback and stay engaged.

The Miro activity used a cohort project spotlight and asked participants to suggest iterations from a DevRel perspective.

## Activity prompts

The activity centered on four questions:

1. What other parts are still needed for this product?
2. What are potential ways to get feedback for this product?
3. How do people learn about the product?
4. How do you attract people to use the product?

## My participation context

I missed the live Miro activity and only reviewed the end state afterward.

Because my original project reached a shelving decision this week, my activity response is retrospective rather than a live build iteration. Any answers to the activity prompts are from the perspective of what the **Open Source Readiness Field Guide** became in Week 03: not a product to continue independently, but a project idea that should be redirected toward an existing ecosystem.

## Hindsight answers for the Open Source Readiness Field Guide

| Prompt | Hindsight answer | What this revealed |
| --- | --- | --- |
| What other parts are still needed? | The most important missing part is not another feature or guide section. It is a map of what OSILK already covers and where contribution gaps may exist. | Before building new material, I need to understand the existing ecosystem deeply enough to avoid duplication. |
| How could I get feedback? | Feedback should come from the existing OSILK maintainers, open source office practitioners, SME stakeholders and people who would actually use open source readiness material. | The feedback path should move from imaginary users toward people already working in the problem space. |
| How do people learn about it? | For this topic, people should likely learn through the existing OSILK project, Eclipse Foundation channels, OSPO networks and practitioner communities rather than through a new standalone field guide. | Discovery could have a stronger path through an existing project and network. |
| How do I attract people to use it? | I should not try to attract people to a competing project. A better route is to contribute, share useful learning and help create gravity around the existing project. | DevRel value can come from strengthening an ecosystem, not only owning a new artifact. This is especially true as OSILK did NOT come up in any of my searches but later on through a personal network connection. |

## What changed or clarified my thinking

The activity questions made the pivot clearer because they pushed me to think about the complete experience around the project.

If I had answered only "What content should I create?", the **Open Source Readiness Field Guide** could possibly still seem like a good next step.

But when I asked:

* How will people discover it?
* Who will trust it?
* Where will feedback come from?
* What makes it more valuable than existing resources?
* What gravity can it create?

…the independent build path looked **weaker**. Especially from a network effect where there is a project out there that has the Eclipse Foundation backing.

## Connection to the pivot project

The same activity prompts are useful for the next project candidate, **PlainTrack**.

| Prompt | Early PlainTrack answer |
| --- | --- |
| What other parts are still needed? | A small MVP, example file format, README, short workflow demo and a simple explanation of when this is useful. |
| How could I get feedback? | Direct feedback from potential users, compliance stakeholders and people who already keep work-time records in organizational tools. |
| How do people learn about it? | Through a short public project page, GitHub README, example workflow and direct sharing with a small known audience. |
| How do I attract people to use it? | By showing a concrete workflow that feels closer to how developers already work, instead of positioning it as another heavy time-tracking platform. |

## Open questions

* Is PlainTrack narrow enough for the remaining DevRel University timeline?
* Is the pain strong enough for users to try a new workflow?
* What is the smallest useful demo that can test the idea without overbuilding?
* Can the project be explained in one clear page?

## Takeaway

The activity helped reframe DevRel work as more than project promotion.

A project needs supporting systems: onboarding, feedback, education, distribution and retention. But before building those systems, I need to know whether the project should exist independently at all.
