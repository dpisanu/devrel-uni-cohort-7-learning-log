One thing I am re-learning while working on 𝗣𝗹𝗮𝗶𝗻𝗧𝗿𝗮𝗰𝗸:
1️⃣ Adding options is easy, too easy.
2️⃣ Protecting the core workflow is harder.

In the last posts I wrote about:
➖ creating a useful trail of work
➖ connecting documentation and measurement
➖ how documentation degrades into rough time'stimates when work gets busy

That 𝚕̲𝚊̲𝚜̲𝚝̲ point keeps coming back to me.
Because when a workflow already breaks under pressure, the answer is should never be: "Add more ..."
▪️"fields"
▪️"categories"
▪️"states"
▪️"rules"

Those things are exactly what makes the workflow harder to maintain.

I am aware that not every work environment looks the same though.
I know reporting requirements are not simple.

The goal 🥅 though is to keep documentation close to the work itself, meaning tooling has to resist becoming a giant option jungle.

There are many things 𝗣𝗹𝗮𝗶𝗻𝗧𝗿𝗮𝗰𝗸 will not be touching with a 10 foot pole:
🔴 approval workflow software
🔴 employee monitoring 
🔴 universal time tracking platform

Right now I am very interested in a smaller question: ❔ What is the 𝘴𝘪𝘮𝘱𝘭𝘦𝘴𝘵 𝘶𝘴𝘦𝘧𝘶𝘭 𝘱𝘢𝘵𝘩 from documented work 𝘵𝘰 𝘢 𝘳𝘦𝘢𝘥𝘢𝘣𝘭𝘦 𝘳𝘦𝘱𝘰𝘳𝘵 ❔

So this part of the capstone is scratching that very personal itch without getting lost in trying to support every workflow.

For 𝗣𝗹𝗮𝗶𝗻𝗧𝗿𝗮𝗰𝗸, I am 𝗱𝗲𝗹𝗶𝗯𝗲𝗿𝗮𝘁𝗲𝗹𝘆 starting with an 𝗼𝗽𝗶𝗻𝗶𝗼𝗻𝗮𝘁𝗲𝗱 shape:
📝 Plain text as the input
🧭 A small set of configurable rules
📊 A readable report as the output

From there on it is about learning which constraints are useful, which ones are limiting and which ones protect the core idea.
