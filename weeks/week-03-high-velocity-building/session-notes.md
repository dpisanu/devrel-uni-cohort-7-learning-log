# Session Notes: Building and Shipping at High Velocity in the Age of AI

## Session metadata

Session: **Building and Shipping at High Velocity in the Age of AI**  
Date: 12.05.2026  
Speaker: Patrick Skinner  
Role: General Manager, Superbuilders

## One-sentence summary

AI-native learning systems for builders are not about replacing understanding; they are about using structured research, feedback loops and ecosystem awareness to learn and ship faster without losing judgment.

## Theme

The official session theme asked what it means to rebuild how people learn and how that changes the way we build and ship.

My framing of the theme:

> This session explored how AI changes learning from a passive content-consumption activity into an active system of research, synthesis, feedback and adaptation — and how that same learning system changes the way we build, ship and stay relevant in fast-moving technical ecosystems.

## Key takeaways

### 1. Understanding remains the builder's moat

AI can help with research, synthesis, outlining, summarizing and workflow automation. But the session repeatedly returned to the idea that understanding cannot be delegated.

For DevRel and product work, this means the builder still has to understand:

* the problem
* the audience
* the ecosystem
* the tooling
* the tradeoffs
* the impact of the work

AI can accelerate thinking, but it cannot own judgment on behalf of the person building.

### 2. High velocity depends on systems

The session framed speed less as personal hustle and more as system design.

Useful systems include:

* systems for daily research
* systems for challenging assumptions
* systems for deciding what matters
* systems for capturing and reusing knowledge
* systems for getting feedback
* systems for updating development practices

This connects directly to DevRel work because DevRel depends on repeated learning, repeated communication, repeated community touchpoints and repeated feedback loops.

### 3. Builders need focus and peripheral vision

The phrase **"snipers both eyes open"** turns out to be a useful metaphor for balancing focus with situational awareness.

The builder still needs a target, but also needs enough peripheral vision to notice:

* ecosystem changes
* model capability changes
* native feature absorption
* workflow obsolescence
* competitor movement
* new user behavior
* emerging community needs

For my own project, this became especially relevant because the broader ecosystem revealed an existing project that already addressed much of my intended value proposition.

### 4. Research should reach useful saturation

The session introduced **Optimal Knowledge Saturation** as a way to think about the cost of additional research.

The useful question is not "Can I learn more?" The useful question is:

> Is the next unit of learning still worth the time, attention and cost it requires?

For project work, this matters because endless research can delay action. At the same time, too little research can create avoidable duplication or weak strategy.

### 5. DevRel can be a gravity point

The session connected DevRel to ecosystem gravity.

In this framing, DevRel is not only about publishing content or answering questions. It helps create conditions where people are attracted to an ecosystem, understand where they fit and have reasons to stay involved.

That gravity can come from:

* useful learning material
* clear onboarding
* credible subject-matter expertise
* community connection
* feedback loops
* repeatable value
* a visible and trustworthy human presence

## Concepts, frameworks and models

### METT-TC

[METT-TC](https://blog.patrickskinner.tech/mett-tc-for-developer-relations) was presented as a planning framework adapted for DevRel and builder strategy.

In the session context, the working components were:

| Component | Meaning in this context |
| --- | --- |
| Mission | The task and its purpose. |
| Ecosystem | The broader technological environment. |
| Target Market | The developers or users the project aims to attract. |
| Troops | The internal team, collaborators, or support system. |
| Time | Deadlines, market windows and sequencing. |
| Cost | Budget, attention, time and other constraints. |

For my own work, METT-TC is useful because the pivot project needs sharper answers about mission, target users, compliance context and the shortest viable build path.

### Orbit model: Gravity, Love, Reach

The Orbit model describes how a community or ecosystem grows and sustains itself.

| Element | Working interpretation |
| --- | --- |
| Gravity | What attracts people and keeps them close to the ecosystem. |
| Love | The emotional connection, trust and commitment people feel. |
| Reach | How the ecosystem expands visibility and participation. |

This model is useful for thinking about both open source projects and developer-facing products. A project does not only need features. It needs a reason for people to notice it, care about it, return to it and tell others about it.

### Optimal Knowledge Saturation

Optimal Knowledge Saturation is a model for deciding when more research is no longer worth the cost.

The practical interpretation for this week:

* 0 to 90 percent understanding can be fast and resource-efficient.
* 90 to 100 percent understanding can require much more time for smaller gains.
* Builders need to know when to stop researching, run a diagnostic and move into action or decision.

For the Open Source Readiness Field Guide, this model helped frame the shelving decision. Additional independent research would likely produce diminishing returns compared with joining or supporting the existing OSILK ecosystem.

## Q&A themes that extended the keynote

The Q&A expanded the keynote into practical workflows:

* using scheduled AI tasks for daily research and summaries
* building a second brain or knowledge system without letting it become the real work
* using research workflows to challenge assumptions
* analyzing value propositions and pain points before building
* thinking about TAM and ICP before deciding whether a project is worth pursuing
* using go-to-market engineering to design distribution and feedback loops
* preserving human voice and judgment in AI-assisted work

The most relevant Q&A idea was the reminder to pull the easiest and most interesting thread when a project becomes too broad or chaotic.

## My interpretation

The session landed at exactly the right moment. I was already realizing that my original project might be too broad and too close to existing work.

The Week 03 concepts gave me language for what was happening:

* **System imperative**: use reflection and decision systems rather than emotion alone.
* **Optimal knowledge saturation**: stop researching the same broad space once the next useful move is clear.
