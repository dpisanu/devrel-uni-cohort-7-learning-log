# Project Brief: Week 03

## Project status

Project name this week: **Open Source Readiness Field Guide: the pivot point**  
Previous direction: Open Source Readiness Field Guide  
Current status: **Shelved as an independent project**  
Emerging direction: **PlainTrack**  
Weekly status: Pivoting from broad enablement resource toward a smaller product-shaped project

## One-sentence project update

This week clarified that the **Open Source Readiness Field Guide** should not become a competing standalone project because an existing Eclipse project, OSILK, already addresses much of the same open source and InnerSource learning space.

## What changed this week

Through a conversation with a friend from the Eclipse Foundation, I learned about [Eclipse OSILK](https://projects.eclipse.org/projects/technology.osilk): the Open Source & InnerSource Learning Kit.

**OSILK** is already focused on modular training material and resources around Open Source and InnerSource. Its scope overlaps strongly with the value proposition I had been exploring for the **Open Source Readiness Field Guide**.

After a presentation and conversation with the **OSILK** project lead, I decided not to continue building a parallel resource. The better decision is to shelve the independent project idea, study what already exists and see whether I can contribute to the existing project or help strengthen its surrounding DevRel ecosystem.

The active call for contribution from **OSILK** made it so much easier.

## Decision made

I decided to shelve the **Open Source Readiness Field Guide** as an independent DevRel University build project.

This does not mean the idea was wrong. It means the ecosystem already has a stronger gravitational center than the one I would be able to create within the remaining time of the program.

The decision is based on three observations:

1. The problem space is still valuable.
2. The independent build path is larger than expected.
3. An existing project already has maturity, structure and people around it.

## Why this is not a failure

The Week 02 session introduced a useful lens for this moment: sunk cost fallacy.

By Week 03, I had invested time into the **Open Source Readiness Field Guide**, but not enough to justify ignoring new evidence. Knowing what I know now, I would not choose to create a competing resource from scratch. That makes shelving the project a responsible decision rather than a failure of execution.

The work done so far is still useful. It helped clarify the audience, problem space, open source readiness concerns and the kinds of enablement material small and midsize companies may need.

## Connection to Week 03 session concepts

### Snipers both eyes open

The project pivot is an example of keeping both eyes open.

One eye stayed focused on the original project idea. The other eye noticed the broader ecosystem: existing projects, potential collaborators, maintainers and the risk of duplicating effort. (even if the eye had to be guided to seeing that specific target)

### The system imperative

The decision also showed the value of having a system for reflection and project review. Without a reflection point, it would have been easier to continue because the idea felt meaningful but sink more time into a project that will continue to play catch up to **OSILK**.

### Optimal knowledge saturation

The **Open Source Readiness Field Guide** reached a point where further independent research would likely produce diminishing returns. The next useful step is no longer "learn more before building." It is either contribute to **OSILK** or redirect build energy into something smaller and testable.

### DevRel as gravity

Instead of trying to create a new gravity point, I can explore how to support the gravity of an existing open source learning ecosystem.

## Emerging pivot project: PlainTrack

The next candidate project is **PlainTrack**.

Working problem statement:

> Work-time records are organizationally important, but most tools make the source of truth opaque, platform-dependent and disconnected from developer workflows. PlainTrack solves the gap between relevant work-time bookkeeping and developer-native, file-based workflows.

This is still early and needs validation, but it appears more product-shaped than the **Open Source Readiness Field Guide**.

## Initial audience hypothesis

The immediate audience hypothesis is a reachable group of roughly 1,000 possible users, with about 300 potential strong early adopters.

The next validation step is to challenge the idea with:

* potential users
* compliance stakeholders
* people who already manage or report work-time records
* developers who prefer file-based or Git-native workflows

## METT-TC sketch for the pivot project

| Area | Current working assumption |
| --- | --- |
| Mission | Build a small, developer-native way to keep work-time bookkeeping close to the source of truth. |
| Ecosystem | Existing time-tracking tools are often platform-centered rather than workflow-centered. |
| Target market | Developers or technical teams that already work in text files, Git, issues, commits and lightweight automation. |
| Troops | Solo builder for now, with possible feedback from users, compliance stakeholders and the DevRel University cohort. |
| Time | The remaining DevRel University timeline requires a narrower MVP and faster feedback cycle. |
| Cost | Keep the scope small enough to test without heavy infrastructure, long research cycles, or complex integrations. |

## What I worked on this week

* Revisited an older tool idea that had been sitting in my backlog.

## Blockers and risks

* The new project needs validation from possible users.
* The compliance angle needs careful handling.
* The MVP must stay small enough for the remaining program timeline.
* I need to avoid turning another focused product idea into a broad research project.

## Decisions postponed

No major decisions are formally postponed yet, but the following need validation in Week 04:

* whether the target users actually feel the problem strongly enough

## Retrospective

### What worked

I was able to pivot without getting trapped by the sunk cost fallacy.

### What felt unclear

It felt unclear whether pivoting this late in a five-week challenge was the right decision.

### What I would change next week

Bring the pivot project up to a concrete enough shape that I can start building instead of only researching.

### What I learned about the project

The **Open Source Readiness Field Guide** is still a good idea, but it is several sizes too large for this short program. It has too many unknowns, too many moving parts and too much overlap with an already maturing project.

The next project should be smaller, sharper and closer to something I can ship.
